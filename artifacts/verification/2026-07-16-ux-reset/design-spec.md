# PitchFlow UX reset design spec

## Accepted internal concept set

- `concepts/entry-desktop-1440x1000.png`
- `concepts/entry-mobile-390x844.png`
- `concepts/direction-desktop-1440x1000.png`
- `concepts/results-desktop-1440x1000.png`

These concepts are implementation references for the user-requested state model. They are not product assets and may not replace real UI, real captures, or generated campaign output.

## State and copy lock

1. Entry: `Paste your repo. Get the whole launch kit.` One repository field, one primary `Generate launch kit` button, one subordinate `Explore the PitchFlow demo` action, and the five labels Website, Images, Videos, Copy, ZIP.
2. Project wizard: Repository, Direction, Engine, Generate. Only the active state owns the main content and primary action.
3. Results: Website, Images, Videos, Copy, Export. Demo results say `Read-only demo · generated from the PitchFlow repository`; fresh results say they were generated by the connected local engine.
4. Evidence is a quiet secondary link. Build Week, hashes, bridge internals, and security prose never enter the primary hierarchy.

## Design system

- Background: charcoal/off-black `#080b09`, never pure black.
- Raised preview surface: `#0d1210`; open sections otherwise use the page background.
- Text: warm white `#f2f4ef`; secondary `#9ca39d`; tertiary `#7c857e` after the AA contrast repair.
- Accent: one flat muted mint `#94e7a1`; no gradient or glow. The generated concept's slight button gradient is rejected as off-spec and will be implemented flat.
- Borders: single-pixel `rgba(242, 244, 239, 0.12)` hairlines. Cards are reserved for actual preview/media elevation.
- Typography: existing Geist-like system sans stack for display and controls; mono only for URLs, commands, and utility labels.
- Display: desktop 64–72px with tight leading; mobile 44–52px. Control text is explicit and never browser-default.
- Spacing: 24/32/48/72/96 scale. Entry is asymmetric at desktop and single-column below 760px.
- Radius: 6px controls, 10px preview surfaces. Avoid giant rounded shells.
- Motion: opacity/transform only, 180–260ms; disabled under `prefers-reduced-motion`.
- Icons: no new icon dependency. Use text labels and a small shared set of production-quality inline SVG arrows/media glyphs only where meaning requires them.

## Component and ownership inventory

- `ProductEntry`: first-viewport promise, repository form, demo action, deliverable rail.
- `ProjectShell`: quiet header, four-step navigation, exit action, active wizard screen.
- `RepositoryStep`: repository identity/validation summary and continue action.
- `DirectionStep`: concise defaults, progressive advanced disclosure, attributed capture editor.
- `EngineStep`: disconnected/connected/policy-fallback state only when required.
- `GenerateStep`: real stage progress, cancellation, retry, errors.
- `ResultsWorkspace`: project identity, read-only/fresh ownership, tabs, real campaign previews, ZIP action.
- Existing intake, bridge, pairing, job polling, asset retrieval, campaign schema, capture validation, and export handlers remain the source of truth.

## Fidelity exceptions required by product truth

- Concept placeholder capture imagery is not shipped; the implementation uses user-provided captures or verified package assets only.
- Concept website chrome is not reconstructed; the implementation presents the real generated website manifest and links the exact generated site asset.
- Entry preview illustrations are code-native abstract deliverable silhouettes, not fake product UI.
